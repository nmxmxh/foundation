# Foundation Distribution, Agent Bundle, and Licensing

Status: implemented baseline
Date: 2026-06-29
Owner: Platform Architecture

For concept definitions, see `foundation_glossary.md`.
For scaffold ownership modes, see `scaffold_manifest.md`.

## Purpose

Foundation adoption must start from a package boundary, not a copied folder.
The distribution contract has three layers:

1. a global Ovasabi CLI that initializes and updates generated projects;
2. registry-published platform modules for Go, TypeScript, and Rust;
3. an agent-native development bundle and licensing engine that keep local
   workspaces compliant without requiring manual setup.

The local CLI now wraps the same manifest and verification paths. The npm
package skeleton under `cmd/ovasabi/package.json` exposes `@ovasabi/cli` by
running the Go CLI source through a Node shim. Prebuilt binary packaging, signed
remote template downloads, and registry publishing remain release-engineering
work.

## Layer 1: Distribution And Onboarding

The target bootstrap command is:

```bash
npx -y @ovasabi/cli init --profile=performance --name=trader_os
```

The implemented CLI lives under `cmd/ovasabi`. It owns these steps:

1. Resolve the requested Foundation profile: `lite`, `core`, `performance`, or
   `regulated`.
2. Verify the developer license when a key or signed license file is supplied.
3. Download a signed template blueprint from the secure Core repository
   (pending; local checkouts use the current repository).
4. Apply `templates/scaffold.manifest.tsv` exactly; file ownership modes remain
   `create`, `force`, and `overwrite`.
5. Initialize `go.work`, `package.json`, lockfiles, Cargo workspaces, and the
   `.foundation` profile marker.
6. Configure package registries and authentication only through local developer
   config or environment variables (pending registry publisher integration).
7. Generate the agent config bundle: `AGENTS.md`, `.cursorrules`,
   `.clauderules`, `CLAUDE.md`, and `.agents/*`.
8. Run the baseline verification suite before reporting success.

Compatibility fallback:

```bash
./init.sh trader_os --profile performance
make foundation-update
```

Fallback scripts must call the same manifest, patch, and check scripts as the
CLI. They may not create a second scaffold contract.

Current local usage:

```bash
cd cmd/ovasabi
go run . init --profile=performance --name=trader_os --skip-license
go run . update --project-dir=../trader_os_v1 --skip-license
go run . refresh --project-dir=../trader_os_v1 --skip-license   # reconcile from declared .foundation state; no overrides
go run . refresh --project-dir=../trader_os_v1 --skip-license --acknowledge-seed-drift
go run . license verify --offline-license --license-file=ovasabi.lic --license-public-key="$(cat ovasabi.pub)"
go run . doctor            # preflight: verifies git/go/node/npm (required) and docker/cargo (lane-optional)
go run . doctor --json     # machine-readable output for agents and CI
npm exec -- ovasabi --help
```

## Registry Package Boundaries

Generated projects consume Core through standard registries:

| Language | Package boundary |
| --- | --- |
| Go | `github.com/ovasabi/foundation/server-kit` |
| TypeScript | `@ovasabi/runtime-transport`, `@ovasabi/frontend-kit`, `@ovasabi/ui-minimal`, `@ovasabi/runtime-native` |
| Rust | `ovasabi-runtime-sdk`, `ovasabi-runtime-native` |

Commercial subscribers use Ovasabi private registries. Registry credentials are
developer-local and must never be committed:

- global config: `~/.ovasabi/config.json`;
- environment: `OVASABI_LICENSE_KEY` and registry-specific token variables;
- package-manager auth files generated by the CLI with project-safe scopes.

Generated projects may still carry a read-only `foundation/` reference during
the migration period for docs, checks, and local development, but application
code must import package boundaries rather than raw `foundation/*/ts/src` or
internal Go paths.

## Layer 2: Agent-Native Config Bundle

`ovasabi init` installs the DevEnv config bundle so Cursor, Claude, Codex,
Copilot, Windsurf, and VS Code agents start with the same contract:

- `AGENTS.md`: universal operating baseline and Definition of Done;
- `.cursorrules`: compact Cursor workspace rules;
- `.clauderules` and `CLAUDE.md`: Claude-compatible workspace rules;
- `.agents/DOMAIN_GUIDE.md`: project-specific domain ownership notes;
- `.agents/POST_INIT.md`: post-bootstrap checklist.

Every agent-facing file must tell the assistant:

1. it is operating in an Ovasabi Foundation workspace;
2. it must inspect `templates/scaffold.manifest.tsv` before editing scaffolded
   files;
3. it is bound by the CP and TE practices, verified through Foundation checks;
4. it should begin with `docs/foundation/foundation_glossary.md` and retrieve
   lane docs only when needed.

The bundle is scaffolded as `create` mode. Project teams may add local domain
guidance, while Foundation updates preserve the root contract through managed
patches and `check-agent-contract`.

## Layer 3: Licensing Engine

License verification must be lightweight, auditable, and friendly to both
cloud-connected startups and air-gapped enterprises.

### Online Verification

For connected commercial users, the CLI reads a key from:

1. `OVASABI_LICENSE_KEY`;
2. `~/.ovasabi/config.json`.

Update commands verify the license before downloading private Core patches when
a license key is supplied:

```text
GET https://api.ovasabi.studio/v1/license/verify
```

The request is bounded by an explicit timeout, includes a CLI-generated request
correlation ID, and avoids sending project source, tenant data, secrets, or
package-lock contents. On success, the CLI accepts an active response; registry
scopes, enabled profiles, expiration, and update channel metadata are planned
extensions.

### Air-Gapped License

Enterprise buyers may use a signed `ovasabi.lic` file. The CLI validates it
offline using an Ed25519 public key supplied by flag or
`OVASABI_LICENSE_PUBLIC_KEY` and checks:

- signature validity;
- issuer and audience claims are parsed for future policy use;
- expiration and not-before timestamps;
- organization identifier;
- non-negative seat count;
- enabled profiles such as `regulated` and `performance`;
- allowed package/update channels are parsed for future policy use.

The offline path may install from an enterprise mirror or a pre-approved
artifact bundle, but it must preserve the same manifest, package-boundary, and
verification contracts as the online path.

## Security And Failure Rules

1. License keys, registry tokens, signed license files, and downloaded
   blueprints are untrusted inputs until verified.
2. Downloaded templates and packages must be integrity checked before scaffold
   application.
3. License verification failure must stop private package sync but leave the
   existing workspace untouched.
4. Network calls must be bounded and must not run during air-gapped mode.
5. The CLI must record only non-secret evidence: profile, package versions,
   verification command results, correlation ID, and update channel.
6. Public, open-source package installs must remain possible without commercial
   registry credentials when the selected profile allows it.

## Required Evidence

CLI and licensing changes require:

- scaffold init/update tests proving manifest modes are preserved;
- package-boundary checks proving generated projects use registry imports;
- agent contract checks proving `AGENTS.md`, `.cursorrules`, `.clauderules`,
  and `CLAUDE.md` are present and point to the local docs;
- online license verifier tests for active, expired, malformed, revoked, and
  timeout responses;
- offline license verifier tests for signature failure, expired claims, seat
  overflow, unsupported profile, and valid air-gapped operation.
